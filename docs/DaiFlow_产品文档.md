# DaiFlow 产品文档

> 版本：v0.1 MVP
> 更新时间：2026-03-11

---

## 一、产品概述

DaiFlow 是一个面向全栈开发者的本地 AI 编程工作台。核心目标是把"从需求到 MR"的完整开发流程产品化，让 AI 真正理解项目上下文，辅助开发者高效完成需求开发。

**一句话概括**：让 AI 理解你的项目，然后从需求描述直接写到代码。

### 产品形态

- 本地软件，通过命令行启动后，在浏览器中访问 Web 界面
- 安装后在本地创建 `.daiflow` 文件夹，存储所有数据和文件

### 核心价值

1. **项目知识沉淀**：AI 自动分析代码仓库，生成项目知识，团队共享，避免每次重复"教 AI 认识项目"
2. **全流程开发辅助**：从技术方案 → 任务拆解 → 编写代码 → 提交 MR，AI 参与每个环节
3. **知识飞轮**：每次开发完成后知识库持续更新，AI 越用越准

## 二、本地文件结构

```
~/.daiflow/
├── daiflow.db                    # SQLite 数据库，存储结构化数据
└── projects/                     # 项目文件目录
    ├── {project_id}/             # 每个项目的文件目录
    │   ├── skills/               # 该项目的 skill 文件
    │   │   ├── {从skill中心拉取的skill}/
    │   │   └── {AI生成的skill}/
    │   └── code/                 # clone 下来的代码（可选）
└── tasks/                        # 任务文件目录
    └── {task_id}/                # 每个任务的文件目录
        ├── .cody/
        │   └── skills/           # 该任务使用的 skill（从项目 skills 同步过来）
        ├── plan.md               # 技术方案文档
        └── todo.json             # 任务拆解列表
```

## 三、核心功能模块

### 3.1 项目管理

#### 功能描述
用户可以在 DaiFlow 中创建和管理项目。项目是 DaiFlow 的最小管理单元，对应一个完整的业务域（如 ByteWorks 中的 cost 模块）。

#### 项目配置信息

| 字段 | 说明 |
|------|------|
| 项目名称 | 项目的名称 |
| 项目描述 | 对项目的文字描述 |
| 仓库列表 | 该项目关联的代码仓库，支持多个 |
| Skill 名称列表 | 需要从 Skill 中心拉取的 skill 名称 |

#### 仓库配置信息

每个仓库需要配置：

| 字段 | 说明 |
|------|------|
| Git 地址 | 远端仓库地址 |
| 本地路径 | 本地已 clone 的代码路径，用户填写 |
| 仓库类型 | 前端仓库 / 后端仓库 / 自定义（可手动输入类型名称） |
| 仓库介绍 | 对该仓库的描述，帮助 AI 理解该仓库的定位和用途 |

#### 项目初始化流程

用户点击保存后，系统自动触发项目初始化：

1. **拉取 Skill**：根据配置的 skill 名称，从远端 Skill 中心拉取对应的 skill 文件，存入 `projects/{project_id}/skills/`
2. **AI 分析生成知识**：AI 扫描项目关联的代码仓库，自动生成项目知识，以 Claude Skill 规范的 Markdown 文件形式沉淀，存入 `projects/{project_id}/skills/`

AI 生成的知识内容包括：
- **项目结构说明**：对前端和后端的目录结构进行解析，说明各目录/模块的职责和组织规范
- 项目模块划分及各模块说明
- 各模块的业务流程
- 前端组件结构和复用情况
- 各模块的数据实体和数据流
- 前后端接口交互关系
- 各模块对下游的依赖

> 每个知识点独立一个 Cody session 并发执行，支持局部更新。

### 3.2 任务管理

#### 功能描述
任务（Task）代表一次具体的需求开发，是启动本地开发流程的基本单元。

#### 任务配置信息

| 字段 | 说明 |
|------|------|
| 任务名称 | 本次开发任务的名称 |
| 关联项目 | 选择该任务所属的项目 |
| 任务描述 | 对本次开发内容的描述 |
| 开发分支 | 代码开发使用的 Git 分支名 |
| 产品需求（PRD） | Markdown 格式输入框，可粘贴 PRD / Figma 链接等 |
| 技术方案 | Markdown 格式输入框，可预先填写已有技术思路 |

#### 任务创建后

- **同步 Skill**：将项目下的所有 skill 文件同步到 `tasks/{task_id}/.cody/skills/` 目录，供 Cody AI 自动识别使用
- **切换代码分支**：在各关联仓库的本地路径下，自动执行 `git checkout -b {branch}` 或 `git checkout {branch}`，切换到任务配置的开发分支
- 任务进入开发流程

---

### 3.3 开发流程

开发流程分为四个阶段，页面顶部始终展示当前节点进度。

```
技术方案 → 任务拆解 → 编写代码 → 代码审查 & 提交
```

---

#### 阶段一：技术方案

**触发**：任务创建后自动进入

**AI 行为**：
- 自动扫描关联仓库的代码目录
- 结合用户填写的 PRD、技术方案输入、以及项目 skill 知识
- 生成初版技术方案，写入 `tasks/{task_id}/plan.md`

**页面布局**：
- 左侧：展示 `plan.md` 技术方案内容（实时更新）
- 右侧：与 AI 的对话框，用户可与 AI 讨论、调整技术方案，AI 的修改直接反映到 `plan.md`

**Cody Session**：技术方案阶段与任务拆解阶段共享同一个 `session_id`，保持上下文连续性。

---

#### 阶段二：任务拆解

**触发**：用户确认技术方案后，点击「锁定方案」按钮

**AI 行为**：
- 基于 `plan.md` 和代码目录，将技术方案拆解为有序的 todo 列表
- 写入 `tasks/{task_id}/todo.json`
- 同时将每个 todo 写入数据库 todos 表，记录序号、标题、详细描述、状态等

**页面布局**：
- 左侧：展示 todo 列表
- 右侧：与 AI 的对话框，用户可与 AI 讨论、调整 todo 内容，AI 修改同步到 `todo.json` 和数据库

**Cody Session**：与技术方案阶段共享同一个 `session_id`。

---

#### 阶段三：编写代码

**触发**：用户确认 todo 列表后，点击「执行编码」按钮

**执行方式**：MVP 阶段采用逐步执行模式，用户手动点击每个 todo 触发执行。

**单个 todo 执行流程**：
1. 用户点击某个 todo 的执行按钮
2. AI 根据该 todo 的描述、`plan.md`、代码目录、以及 `.cody/skills/` 下的 skill，开始执行
3. 执行过程中，中间区域实时展示执行进度和输出结果，包括文件改动情况
4. 执行完成后，用户可在右侧对话框与 AI 沟通，补充或调整该 todo 的改动结果
5. 确认无误后，继续点击下一个 todo

**每个 todo 有独立的 `session_id`**，todo 之间通过 `plan.md` 共享上下文。

**页面布局**：
- 左侧：todo 列表，展示每个 todo 的执行状态
- 中间：当前 todo 的执行结果，展示代码改动内容（类似 diff 视图）
- 右侧：与 AI 的对话框

---

#### 阶段四：代码审查 & 提交

**触发**：所有 todo 执行完毕后，用户点击「下一步」进入

**页面布局**：
- 左侧：整体代码变更内容，类似 GitHub MR 的 diff 视图，展示本次开发涉及的所有文件改动
- 右侧：与 AI 的对话框，用户可进行最后的调整

**提交流程**：
1. 用户确认代码无误后，点击「生成 MR」
2. AI 自动生成 commit message
3. 系统自动执行 git commit + push，将代码推送到对应分支

## 四、技术调研

### 4.1 AI 执行内核：Cody SDK

**项目地址**：https://github.com/CodyCodeAgent/cody

Cody 是一个开源 AI 编程助手框架，Python SDK（`cody.sdk`）以 in-process 方式直接包装核心引擎，无需启动额外 HTTP 服务。

**核心能力：**
- 支持 `run()` / `stream()` 两种执行方式，stream 可实时将执行过程推送给前端
- 内置 session 机制：同一 `session_id` 的多次调用共享上下文记忆，天然支持多轮对话
- 支持 `allowed_roots` 配置多工作目录，适合 Monorepo 多仓库场景
- 技能（Skill）自动加载：Agent 启动时自动识别 `.cody/skills/` 目录下的 Skill 文件并注入上下文
- 提供 28+ 个内置工具，包括文件读写、命令执行、git 操作、代码搜索等

**DaiFlow 中的应用：**
- 项目知识库生成：每个知识点独立一个 session 并发执行，提升生成效率，也支持局部更新
- 技术方案阶段与任务拆解阶段共享同一 session_id，保持上下文连续
- 每个 todo 执行拥有独立 session_id，todo 之间通过 plan.md 传递共享上下文
- 通过 `allowed_roots` 配置任务关联的多个本地仓库路径，支持跨仓库代码操作

---

### 4.2 代码 Diff 展示：react-diff-viewer

**npm 包**：`react-diff-viewer`

用于代码审查阶段展示本次开发的所有文件变更，类似 GitHub MR 的 diff 视图。

**核心特性：**
- 支持 split（左右对比）和 unified（单列）两种展示模式
- 支持语法高亮
- 轻量，接入简单

**DaiFlow 中的应用：**
- 后端通过 `git diff` 命令获取本次任务的所有文件变更内容
- 前端通过 react-diff-viewer 渲染展示，供用户在代码审查阶段做最终确认

## 五、MVP 范围

一期 MVP 目标：

1. **项目管理**：项目的创建、编辑、删除；项目初始化（拉取 skill + AI 生成知识）
2. **任务管理**：任务的创建、列表展示、进入任务详情
3. **开发流程跑通**：四个阶段完整可用——技术方案生成与调整、任务拆解与调整、逐步执行编码、代码审查与提交
