# 前端优化讨论

## 1. 侧边栏收起/展开

### 现状分析

当前页面布局采用经典的 `sidebar + main` 结构（[Shell.tsx](../frontend/src/components/Shell/Shell.tsx)）：

```
┌──────────┬─────────────────────────────┐
│ sidebar  │         main content        │
│ (180px)  │         (flex: 1)           │
│          │                             │
│ - 项目    │                             │
│ - 任务    │                             │
│          │                             │
│          │                             │
│ 设置      │                             │
│ 模型状态   │                             │
└──────────┴─────────────────────────────┘
```

- sidebar 固定宽度 180px，无收起功能
- 在 DevFlow 四阶段页面中（Plan/Todo/Coding/Review），内容区本身就需要左中右多栏布局，sidebar 占据的 180px 比较浪费

### 改进方案

在 sidebar 底部（model-pill 上方）或顶部（logo 旁边）添加一个收起/展开按钮：

**收起状态（~52px 宽）：**
- 只显示图标（logo 缩为 "D"，导航项只保留 icon）
- hover 时可 tooltip 显示文字
- model-pill 缩为只显示状态圆点

**展开状态（180px 宽，当前默认）：**
- 与现在一致，显示完整图标 + 文字

**交互细节：**
- 收起按钮位置：sidebar 底部，model-pill 上方，用一个 `«` / `»` 箭头图标
- 过渡动画：sidebar width + 内容 opacity 使用 CSS transition（~200ms）
- 状态持久化：收起/展开状态存入 localStorage，刷新后保持
- 响应式：移动端（<768px）保持现有横向布局，不受影响

**技术实现要点：**

1. Shell 组件新增 `collapsed` state + localStorage 持久化
2. sidebar 添加 `.sidebar.collapsed` CSS class：
   - `width: 52px`
   - `.nav-label`, `.logo-text`, `.model-text` 隐藏（`display: none` 或 `opacity: 0`）
   - `.nav-item` 居中显示，只保留 icon
3. 添加 toggle 按钮组件
4. CSS transition 确保平滑动画

## 2. DevFlow 四阶段统一布局抽象

### 现状分析

四个阶段页面（Plan / Todo / Coding / Review）结构相似但各自实现，没有统一的布局骨架：

| 阶段 | 内容展示区 | 操作按钮 | 聊天区 | 布局方式 |
|------|----------|---------|-------|---------|
| Plan | MarkdownViewer (plan.md) | 确认方案、重新生成 | ChatPanel | ResizableSplitPane |
| Todo | Todo 列表 + 详情弹窗 | 开始编码、重新拆解 | ChatPanel | ResizableSplitPane |
| Coding | Todo 时间线 + DiffViewer | 执行 Todo（内联）、进入审查 | ChatPanel | 三栏 devflow-body |
| Review | Diff 统计 + DiffViewer | 提交 MR | ChatPanel | 两栏 devflow-body |

问题：
- Plan/Todo 用 `ResizableSplitPane`，Coding/Review 用 `devflow-body` flex 布局，不统一
- 操作按钮散落在内容区内部，没有固定位置
- 每个页面都重复写 Topbar + StageProgress + 布局结构

### 改进方案：统一布局骨架 `StageLayout`

将四个阶段抽象为统一的区域划分：

```
┌─────────────────────────────────────────────────┐
│  Title Area — Topbar（任务名、分支、状态）          │
├─────────────────────────────────────────────────┤
│  Progress Area — StageProgress（四阶段进度条）     │
├──────────────────────┬──────────────────────────┤
│  Content Display     │   Chat Area              │
│  (主要内容展示)       │   (ChatPanel)            │
│                      │◄── 中间线可左右拖拽 ──►    │
│                      │   (可收起，缩到最小宽度)    │
│                      │                          │
├──────────────────────┴──────────────────────────┤
│  Action Bar（操作按钮区，横跨底部）                 │
└─────────────────────────────────────────────────┘
```

**各阶段填充内容：**

| 区域 | Plan | Todo | Coding | Review |
|------|------|------|--------|--------|
| Content Display | plan.md 渲染 | Todo 列表 | Todo 时间线 + Diff 查看器 | Diff 统计 + Diff 查看器 |
| Action Bar | 确认方案 / 重新生成 | 开始编码 / 重新拆解 | 进入代码审查 | 提交 MR |
| Chat Area | 方案讨论 | Todo 讨论 | 编码对话 | 审查讨论 |

**Coding 阶段特殊性：** Content Display 内部再分为左侧 Todo 时间线 + 右侧 Diff 查看器，但从布局骨架角度看仍然是一个整体的 Content Display 区域，内部分栏由该阶段自行处理。

### 滚动隔离原则

每个区域有独立的滚动域，一个模块的内容不能把另一个模块撑爆：

- **Title Area / Progress Area**：固定高度，不滚动
- **Content Display**：`overflow-y: auto`，内容超出时独立滚动
- **Action Bar**：固定在内容区底部，不随内容滚动（始终可见）
- **Chat Area**：ChatPanel 已有独立滚动，不受左侧内容影响

实现方式：Work Content Area 用 `flex-direction: column`，Content Display 设 `flex: 1; min-height: 0; overflow-y: auto`，Action Bar 设 `flex-shrink: 0`。`min-height: 0` 是关键，防止 flex 子元素撑破父容器。

### 技术实现要点

1. **新建 `StageLayout` 组件**，接收 props：
   ```tsx
   interface StageLayoutProps {
     taskId: string
     task: TaskData
     currentStage: 1 | 2 | 3 | 4
     content: React.ReactNode      // Content Display 区域
     actions?: React.ReactNode     // Action Bar 区域
     chatTitle: string
     chatMessages: Message[]
     chatOnSend: (msg: string) => void
     chatStreaming: boolean
     chatDisabled?: boolean
   }
   ```

2. **布局统一使用 ResizableSplitPane**（左：content + actions，右：chat），替代 Coding/Review 现有的 `devflow-body` 三栏布局

3. **Action Bar 固定在内容区底部**，使用 `position: sticky; bottom: 0` 或 flex 布局撑底

4. **四个 Stage 页面简化为**：只负责业务逻辑 + 往 StageLayout 里填内容，不再关心整体布局

## 3. 阶段只读锁定

### 规则

当任务进入下一阶段后，之前所有阶段自动变为**只读模式**：

- Content Display：只能查看，不能编辑
- Action Bar：所有按钮禁用（灰态）
- Chat Area：输入框禁用，不能发送消息，历史消息仍可查看

例如：进入 Coding 阶段后，Plan 和 Todo 阶段都变只读。当最后一个阶段（Review）完成后（TaskStatus=DONE），所有四个阶段全部变为只读，整个任务归档。

### 现状

各阶段已有部分锁定逻辑（`isLocked`、`chatDisabled`），但分散在四个页面中各自判断，没有统一收口。

### 改进方案

在 `StageLayout` 骨架层统一处理：根据 `task.status` 和 `currentStage` 计算出 `readonly` 标志，自动禁用 Action Bar 按钮和 Chat 输入，Content Display 由各阶段自行根据 `readonly` prop 处理（如隐藏编辑入口）。各阶段页面不再需要自己算锁定状态。

## 4. Todo 跳过功能（已实现）

### 需求

在 Coding 阶段，允许用户跳过某个 Todo，不执行直接标记为已跳过。跳过是终态，不可恢复。SKIPPED 的 Todo 不阻塞进入 Review 阶段。

### 实现

- **后端**：`TodoStatus.SKIPPED = 4`，新增 `POST /api/todos/{id}/skip` 接口，只允许 PENDING 状态跳过
- **前端**：PENDING 的 Todo 同时显示"执行"和"跳过"按钮，SKIPPED 的 Todo 标题加删除线、灰色图标
- **allDone 判断**：DONE 或 SKIPPED 都算完成
